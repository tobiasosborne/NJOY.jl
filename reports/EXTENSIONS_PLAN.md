# NJOY.jl Extensions Plan: Research Opportunities Enabled by Differentiable Nuclear Data Processing

**Date:** 2026-03-21

## Executive Summary

NJOY.jl -- a differentiable, composable Julia reimplementation of NJOY2016 -- is uniquely
positioned to address six major pain points in the nuclear data community. This report
surveys recent literature (2020--2026) identifying challenges that are fundamentally
difficult with Fortran-based tools but become tractable when the entire processing chain
is pure-functional, AD-compatible, and callable as a library. Each section identifies
specific papers and challenges, explains why current tools fall short, describes how
NJOY.jl's API would help, and estimates impact.

---

## 1. Nuclear Data Uncertainty Quantification

### 1.1 The Challenge

Uncertainty quantification (UQ) for nuclear data is dominated by two paradigms:

- **Total Monte Carlo (TMC):** Generate hundreds of random ENDF files (via TALYS model
  parameter sampling), process each through NJOY, run each through a transport code, and
  compute variance on the output. Developed by Koning and Rochman, TMC underpins the
  TENDL library, which ships ~300 random files per isotope for UQ.

- **Covariance Sandwich Rule:** Use NJOY/ERRORR to collapse ENDF MF33 covariance data
  to multigroup, then propagate via `Cov_response = J * Cov_params * J'` where `J` is a
  sensitivity Jacobian. Tools like SANDY (Fiorito et al.), SHARK-X (PSI), and
  SCALE/TSUNAMI (ORNL) implement variants of this workflow.

**Key papers and projects:**

- Biot, Ducru, Sobes, Lewis, and Forget, "Extending Embedded Monte Carlo as a novel
  method for nuclear data uncertainty quantification," *EPJ Nuclear Sciences &
  Technologies* (2025). The EMC method embeds UQ within a single Monte Carlo simulation
  by treating each batch as a new random sample, reducing TMC's requirement for hundreds
  of independent transport runs.

- Vaara, "Nuclear Data Uncertainty Propagation with Total Monte-Carlo Method," Aalto
  University thesis (2020+). Documents the full TMC pipeline and its computational cost:
  each of ~300 random ENDF files must be separately processed through NJOY's
  RECONR-BROADR-ACER chain before transport.

- Fiorito et al., "Nuclear data uncertainty propagation to integral responses using
  SANDY," *Annals of Nuclear Energy* (2017+, actively developed through 2025). SANDY
  samples from multivariate normal distributions using MF33 covariances, writes
  perturbed ENDF files, and calls NJOY externally.

- Ferrer et al., "Nuclear Data Uncertainty Methods in CASMO5 using ENDF/B-VIII
  Covariances," (2024). Multigroup covariance libraries generated by NJOY/ERRORR are
  propagated through lattice physics; processing is a one-time but fragile step.

- Rochman and Koning, "Full Model Nuclear Data and Covariance Evaluation Process Using
  TALYS, Total Monte Carlo and Backward-forward Monte Carlo," *Nuclear Data Sheets*
  (2014, methodology still in use through TENDL-2023).

### 1.2 Why Fortran NJOY Makes This Hard

1. **TMC requires N separate NJOY runs.** Processing 300 random ENDF files through the
   RECONR-BROADR-THERMR-ACER chain for a single isotope takes hours. For a full library
   (hundreds of isotopes x 300 files x multiple temperatures), the cost is enormous.

2. **NJOY is a black box for gradients.** The sandwich rule `J * Cov * J'` requires
   the Jacobian `d(sigma_g)/d(params)`, which NJOY.jl's ERRORR currently computes by
   *finite differences* (see `sensitivity_jacobian` in `errorr.jl`). Fortran NJOY
   cannot provide exact gradients at all.

3. **SANDY/SHARK-X must shell out to NJOY.** These Python/Perl tools write perturbed
   ENDF files to disk, invoke NJOY as a subprocess, parse the output, and repeat. The
   file-I/O and process-launch overhead dominates wall time.

4. **Covariance processing is fragile.** NJOY/ERRORR's handling of MF33 LB-types,
   mixed absolute/relative blocks, and cross-reaction correlations requires "particular
   care" (NEA WKP(2023)1) -- different processing choices lead to "dramatic differences
   in end results."

### 1.3 How NJOY.jl Makes This Easy

**Exact gradients via AD:**

```julia
using ForwardDiff, NJOY

# Sensitivity of group-averaged cross sections to resonance parameters
function sigma_g(params)
    # Perturb resonance widths, reconstruct, broaden, group-average
    xs = cross_section_slbw(E, modified_params, range)
    group_average(energies, xs, LANL_30; weight=inv_e_weight)
end

# Exact Jacobian -- no finite differences needed
J = ForwardDiff.jacobian(sigma_g, params)
# Sandwich rule with exact J
cov_response = sandwich_covariance(J, cov_params)
```

Because every function in NJOY.jl -- `cross_section_slbw`, `cross_section_rm`,
`doppler_broaden`, `group_average`, `sigma1_at` -- is pure, mutation-free, and
parameterized on element type, ForwardDiff.jl propagates dual numbers through the
entire chain without modification.

**In-memory TMC without file I/O:**

```julia
# Process 300 random parameter sets in-memory, no disk I/O
results = map(1:300) do i
    params_i = sample_resonance_params(cov_matrix)
    pendf = reconstruct_from_params(params_i; err=0.001)
    broadened = doppler_broaden(pendf, 600.0; awr=236.0)
    group_average(broadened.energies, broadened.xs[:,1], LANL_30)
end
mean_xs = mean(results)
std_xs  = std(results)
```

No subprocess launches. No temporary files. Trivially parallelizable with
`Threads.@threads` or `Distributed.pmap`.

**Estimated impact:** HIGH. Exact gradients eliminate finite-difference noise in
sensitivity studies and reduce TMC wall time by 10--100x through in-memory processing.

## 2. Machine Learning for Nuclear Data

### 2.1 The Challenge

Machine learning is increasingly used to predict, interpolate, and evaluate nuclear
cross sections, but current approaches are disconnected from the physics of the
processing chain.

**Key papers:**

- Lovell et al., "Learning nuclear cross sections across the chart of nuclides with
  graph neural networks," *Physical Review C* (2024, LLNL-JRNL-2011023). Uses
  variational autoencoders (VAE) and implicit neural representations (INR) to encode
  cross sections into latent space, then graph neural networks (GNN) to predict missing
  isotopes. The GNN leverages the topological structure of the nuclear chart. Notably,
  clustering on the latent space rediscovers neutron magic numbers.

- Papadimitriou, Sobes et al., "Enhancing nuclear cross-section predictions with deep
  learning: the DINo algorithm," *European Physical Journal Plus* (2025). DINo learns
  correlations between charge-changing and total cross sections from TENDL-2021.

- Khan et al., "Bayesian Learning of (n,p) Reaction Cross Sections with Quantified
  Uncertainties," arXiv:2603.04789 (2025). Bayesian neural networks trained on
  ENDF/B-VIII.1 (n,p) data with uncertainty quantification.

- Kim et al., "Experimental data-driven modeling and prediction of (gamma,n) cross-
  sections with physics-informed neural networks," *Nuclear Instruments and Methods B*
  (2025). PINNs fit EXFOR experimental data while enforcing energy thresholds and
  resonance behavior as constraints.

- Alhajri et al., "Evaluating machine learned nuclear data precision in full core
  nuclear reactor Monte Carlo neutronics," *Scientific Reports* (2026). Evaluates
  ML-compressed nuclear data in full-core OpenMC simulations, showing acceptable
  accuracy with significant speedup.

### 2.2 Why Fortran NJOY Makes This Hard

1. **No differentiable path from resonance parameters to observables.** ML models
   predict raw cross sections or resonance parameters, but there is no way to
   backpropagate a loss through the NJOY processing chain. Training a neural network
   to predict resonance parameters that, after processing, match integral benchmarks
   is impossible without a differentiable processing code.

2. **Physics constraints cannot be enforced during training.** The PINN approach adds
   physics as soft constraints in the loss function, but the processing physics
   (Doppler broadening, self-shielding, group averaging) is outside the ML framework.

3. **Validation requires an expensive round-trip.** To verify that ML-predicted cross
   sections are acceptable, they must be written to ENDF format, processed through
   NJOY, and run through a transport code -- a cycle that takes minutes per iteration.

### 2.3 How NJOY.jl Makes This Easy

**Physics-informed loss function:**

```julia
using Flux, NJOY

# Neural network predicts resonance parameters from (Z, N)
model = Chain(Dense(2, 64, relu), Dense(64, n_params))

function physics_loss(Z, N, target_xs)
    params = model([Z, N])
    # Differentiable processing chain as physics constraint
    xs = cross_section_slbw(E_grid, params_to_slbw(params), range)
    broadened = doppler_broaden(E_grid, xs, 300.0; awr=awr)
    grouped = group_average(broadened..., LANL_30)
    sum((grouped .- target_xs).^2)
end

# Gradient flows through NJOY.jl's processing chain into the neural net
Flux.train!(physics_loss, Flux.params(model), data, Adam())
```

**Hybrid ML-physics evaluation:**

```julia
# Use ML model for fast prediction, NJOY.jl for verification
predicted = ml_model(Z, N)
exact = cross_section_rm(E, exact_params, range)
calibration_error = norm(predicted - exact)
```

**Estimated impact:** HIGH. NJOY.jl enables training neural networks end-to-end
through the processing chain, creating physics-informed ML models that respect Doppler
broadening, self-shielding, and group-averaging physics by construction.

## 3. Reactor Design Optimization

### 3.1 The Challenge

Reactor core design optimization requires evaluating thousands of candidate
configurations, each involving neutronics, thermal-hydraulics, and fuel performance.
Nuclear data processing is a prerequisite for each evaluation.

**Key papers:**

- Yang et al., "Gradient-Informed Design Optimization of Select Nuclear Systems,"
  *Nuclear Science and Engineering* 196(12), 2022. Uses SCALE/TSUNAMI sensitivity
  coefficients for gradient-based optimization of reactor core components, demonstrating
  that gradient-informed optimization scales well with design dimensionality.

- Seurin et al., "AI-based design of a nuclear reactor core," *Scientific Reports*
  (2021). Deep reinforcement learning for flexible-geometry core design achieves 3x
  improvement in temperature peaking factor.

- Yamamoto et al., "Development of core design optimization process: Feasibility study
  of multivariable optimization via integrated sequential analyses," *Annals of Nuclear
  Energy* (2025). JAEA's ARKADIA framework for fast reactor optimization using Bayesian
  optimization.

- Kim et al., "Design Optimization of Nuclear Fusion Reactor through Deep Reinforcement
  Learning," arXiv:2409.08231 (2024).

### 3.2 Why Fortran NJOY Makes This Hard

1. **Nuclear data is pre-processed, not part of the optimization loop.** Cross section
   libraries are generated once and assumed fixed during optimization. But optimal design
   depends on the nuclear data (e.g., optimal moderator-to-fuel ratio depends on
   scattering cross sections at the operating temperature).

2. **Gradient-based optimization cannot reach through NJOY.** TSUNAMI provides
   sensitivity coefficients for transport, but the chain ENDF -> RECONR -> BROADR ->
   GROUPR -> transport is not end-to-end differentiable. The Jacobian
   `d(k_eff)/d(resonance_params)` cannot be computed.

3. **Temperature-dependent optimization is expensive.** Changing operating temperature
   requires reprocessing cross section libraries (new BROADR runs for each temperature),
   making thermal-hydraulic feedback loops in optimization prohibitively costly.

### 3.3 How NJOY.jl Makes This Easy

**End-to-end differentiable design objective:**

```julia
using Optimization, NJOY

function design_objective(design_params)
    temperature = thermal_hydraulics(design_params)
    # Reprocess on-the-fly for the actual temperature
    pendf = reconstruct("u238.endf"; err=0.001)
    broadened = doppler_broaden(pendf, temperature; awr=236.0)
    mg_xs = group_average(broadened.energies, broadened.xs, LANL_30)
    return reactivity(mg_xs, design_params)
end

# AD gradient of design objective through the processing chain
grad = ForwardDiff.gradient(design_objective, initial_design)
```

**Fast temperature sweeps:**

```julia
# Sweep operating temperatures without disk I/O
temperatures = 300.0:50.0:1200.0
xs_vs_T = [doppler_broaden(pendf, T; awr=236.0) for T in temperatures]
```

**Estimated impact:** MEDIUM-HIGH. Most reactor design optimization currently treats
nuclear data as fixed inputs. Making the processing chain differentiable and fast
enough for inner-loop evaluation opens new design spaces.

## 4. Nuclear Data Assimilation

### 4.1 The Challenge

Nuclear data assimilation (or adjustment) uses Bayesian methods to update evaluated
nuclear data based on integral experimental results (critical assemblies, reaction rate
measurements, etc.). The goal is posterior nuclear data that simultaneously agrees with
both differential measurements and integral benchmarks.

**Key papers:**

- Salvatores and Palmiotti, tribute article in *EPJ Nuclear Sciences & Technologies*
  (2021). Reviews decades of work on integral experiments and data assimilation.
  Salvatores was instrumental in establishing the combined use of integral and
  differential data in validation. Palmiotti conceived and directed many experimental
  campaigns.

- Rochman et al., "Monte Carlo nuclear data adjustment via integral information,"
  *European Physical Journal Plus* (2018, methodology active through 2023). Three
  Monte Carlo methods -- BMC, BFMC, and MOCABA -- include integral benchmark
  information in the evaluation procedure, producing posterior nuclear data and
  covariances.

- Noguere et al., "Assimilation of integral experiments on high-energy nuclear
  parameters," *EPJ Web of Conferences* (2023, ND2023). Uses Bayesian inference
  with continuous-energy reactivity sensitivities coupled to a stochastic transport
  code and nuclear data evaluation code.

- Hursin et al., "Applying SHARK-X to perform data assimilation with the LWR-PROTEUS
  Phase II integral experiments," *Annals of Nuclear Energy* (2020). SHARK-X (PSI)
  performs sensitivity analysis, UQ, and data assimilation built around CASMO-5.

- Cacuci, "Information content of integral experiments data: A Bayesian approach for
  quality assurance," *Annals of Nuclear Energy* (2021).

### 4.2 Why Fortran NJOY Makes This Hard

1. **The sensitivity chain is broken.** Data assimilation requires
   `d(integral_response)/d(nuclear_params)`, which factors as:
   `d(response)/d(multigroup_xs) * d(multigroup_xs)/d(pointwise_xs) * d(pointwise_xs)/d(resonance_params)`.
   Fortran NJOY cannot provide the latter two Jacobians analytically.

2. **Processing must be repeated for each posterior sample.** Bayesian methods like
   BMC and BFMC generate posterior parameter samples; each must be processed through
   the full NJOY chain to get usable cross sections.

3. **MOCABA linearizes around the prior.** The MOCABA method uses linearized
   sensitivities (computed by finite differences through NJOY), which limits its
   accuracy for large adjustments far from the prior.

### 4.3 How NJOY.jl Makes This Easy

**Exact end-to-end sensitivity:**

```julia
using ForwardDiff, NJOY

# Full derivative chain: resonance params -> pointwise -> broadened -> grouped
function grouped_xs(resonance_params)
    xs_E = cross_section_rm(E_grid, params_to_rm(resonance_params), range)
    broadened_E, broadened_xs = doppler_broaden(E_grid, xs_E, T; awr=awr)
    group_average(broadened_E, broadened_xs, LANL_30; weight=inv_e_weight)
end

# Exact Jacobian for data assimilation
J_processing = ForwardDiff.jacobian(grouped_xs, prior_params)
# Combine with transport sensitivity for full chain
J_total = J_transport * J_processing
# Bayesian update
posterior_cov = inv(inv(prior_cov) + J_total' * inv(exp_cov) * J_total)
posterior_mean = posterior_cov * (inv(prior_cov) * prior_mean +
                                  J_total' * inv(exp_cov) * exp_data)
```

**In-memory posterior sampling:**

```julia
# Generate and process posterior samples without disk I/O
posterior_samples = rand(MvNormal(posterior_mean, posterior_cov), 1000)
processed = [begin
    xs = cross_section_rm(E_grid, sample_to_rm(s), range)
    doppler_broaden(E_grid, xs, T; awr=awr)
end for s in eachcol(posterior_samples)]
```

**Estimated impact:** HIGH. Nuclear data assimilation is one of the most important
activities in the nuclear data community, and exact gradients through the processing
chain would represent a qualitative advance over current finite-difference approaches.

## 5. On-the-Fly Cross Section Evaluation

### 5.1 The Challenge

Traditional Monte Carlo transport codes (MCNP, Serpent, OpenMC) require pre-processed
cross section libraries (ACE format). This creates several problems: libraries must be
generated at discrete temperatures, memory scales with the number of temperatures and
isotopes, and the preprocessing step itself is a workflow bottleneck.

**Key papers:**

- Josey, Ducru, Forget, and Smith, "Windowed multipole for cross section Doppler
  broadening," *Journal of Computational Physics* 307, 715--727 (2016). The WMP method
  stores cross sections as poles in the complex plane, enabling analytical Doppler
  broadening to arbitrary temperature. Implemented in OpenMC, reduces memory by storing
  compact pole data instead of dense energy grids.

- Li et al., "Hybrid windowed networks for on-the-fly Doppler broadening in RMC code,"
  *Nuclear Science and Techniques* (2021). Uses a Back Propagation neural network
  to compress WMP data further, reducing memory by ~65% while maintaining accuracy.

- Yesilyurt et al., "Comparison of Temperature-Dependent Cross Section Treatments
  within the Shift Monte Carlo Radiation Transport Code," (2023, SSRN). Compares
  multiple on-the-fly broadening methods (including multipole) in realistic reactor
  assemblies with continuous temperature profiles.

- OpenMC documentation (current): Supports WMP format for on-the-fly Doppler
  broadening to arbitrary temperature in the 0K--3000K range for total, elastic,
  absorption, and fission reactions.

### 5.2 Why Fortran NJOY Makes This Hard

1. **NJOY is a batch processor, not a callable library.** It reads an ENDF file,
   writes a PENDF/ACE file, and exits. There is no API to evaluate
   `sigma(E, T, material)` at a single energy point.

2. **WMP requires a separate fitting step.** The windowed multipole representation
   requires fitting poles to the R-matrix cross section -- a separate code outside
   NJOY entirely. The conversion chain is ENDF -> NJOY -> ACE -> WMP fitting tool.

3. **Memory-temperature tradeoff is fixed at preprocessing time.** Users must
   decide which temperatures to process before running transport. Adding a new
   temperature requires rerunning the entire NJOY chain.

### 5.3 How NJOY.jl Makes This Easy

**Direct callable cross section evaluation:**

```julia
using NJOY

# Read resonance parameters once
mf2 = read_mf2(open("u238.endf"), mat=9237)

# Evaluate at any (E, T) -- no preprocessing, no files
for E in neutron_energies
    xs = cross_section(E, mf2.isotopes[1].ranges[1])
    # Returns CrossSections(total, elastic, fission, capture)
end
```

The `cross_section(E, range)` function dispatches on formalism type (SLBW, MLBW,
Reich-Moore) via Julia's multiple dispatch, evaluating the R-matrix formula directly.
Combined with `sigma1_at(E, ...)` for Doppler broadening, this gives
`sigma(E, T, material)` callable at any single point.

**Embedding in a transport code:**

```julia
# In a Julia-based transport code's collision kernel:
function sample_collision(particle, material)
    E = particle.energy
    T = local_temperature(particle.position)
    # On-the-fly evaluation -- no lookup table
    xs = cross_section(E, material.resonance_range)
    # Doppler correction at exact local temperature
    xs_broadened = sigma1_at(E, material.pendf_e, material.pendf_xs,
                             material.awr / (PhysicsConstants.bk * T))
    return xs_broadened
end
```

**Continuous temperature fields:**

```julia
# No need to pre-select temperature grid
temperatures = [local_T(r) for r in mesh_points]  # arbitrary distribution
# Each point broadened to its exact local temperature
xs_field = [sigma1_at(E, pendf_e, pendf_xs,
                      awr / (PhysicsConstants.bk * T))
            for T in temperatures]
```

**Estimated impact:** MEDIUM-HIGH. On-the-fly evaluation eliminates the preprocessing
bottleneck for Julia-native transport codes and enables continuous temperature fields
without memory overhead. The impact is medium rather than high because existing
WMP implementations (OpenMC) already address part of this need for established codes.

## 6. Multi-Physics Coupling

### 6.1 The Challenge

Reactor simulation requires coupling neutronics with thermal-hydraulics, fuel
performance, and chemistry. The nuclear data interface -- where processed cross
sections enter the simulation -- is a persistent pain point.

**Key papers:**

- Gaston et al., "Data transfers for nuclear reactor multiphysics studies using the
  MOOSE framework," *Frontiers in Nuclear Engineering* (2025). The NEAMS program
  supports Griffin (neutronics), SAM (systems T-H), Pronghorn (multidimensional T-H),
  and BISON (fuel performance) on the MOOSE platform. Data transfer between codes
  with different spatial discretizations and data structures is a major challenge.

- Song et al., "Multi-physical coupling study of neutronics/thermal-hydraulics/material
  corrosion," *Nuclear Engineering and Design* (2024). Demonstrates the difficulty of
  coupling three physics domains with different data interfaces.

- COMPASS platform for lead-cooled fast reactor safety: achieves dynamic coupling
  between Serpent and SAS4A/SASSYS-1 through semi-implicit Picard iteration (2025).

- IAEA CRP F12028, "Development of neutronic and thermal-hydraulic coupled
  calculational methodologies for research reactors including treatment of
  uncertainties" (launched 2022). International effort to standardize coupled
  calculations with uncertainty propagation.

- Altahhan et al., "JULIA for enhancing nuclear engineering simulations (JENES):
  introduction to the JENES project and platform," IAEA (2022). NCSU project
  demonstrating Julia's viability for nuclear engineering, including a Nodal Expansion
  Method code transferred from Fortran with comparable performance.

### 6.2 Why Fortran NJOY Makes This Hard

1. **Cross sections are files, not functions.** The coupling interface is "neutronics
   code reads ACE/GENDF file produced by NJOY." When temperature changes, either the
   file must be regenerated or the code must interpolate between pre-tabulated
   temperatures -- introducing interpolation error and constraining the temperature grid.

2. **No common data model.** NJOY writes PENDF (sequential card-image format), ACE
   (transport code input), or GENDF (multigroup). Each downstream code has its own
   reader. Coupling two codes that both need nuclear data means processing twice in
   different formats.

3. **Temperature feedback requires outer iteration.** In a coupled neutronics-T/H
   calculation, temperature changes from T/H require new cross sections from
   neutronics. With NJOY in the loop, this means regenerating libraries at each
   Picard iteration -- prohibitively expensive.

4. **Uncertainty propagation across physics is intractable.** Propagating nuclear data
   uncertainty through a coupled neutronics-T/H-fuel simulation requires sensitivity
   of the coupled response to nuclear data parameters -- impossible when NJOY is a
   black-box file converter.

### 6.3 How NJOY.jl Makes This Easy

**Cross sections as Julia functions, not files:**

```julia
using NJOY

# Load once, evaluate anywhere
material = reconstruct("u238.endf")

# In a coupled T-H/neutronics iteration:
function update_neutronics(T_field, material)
    # Rebroaden at each mesh point's local temperature
    [doppler_broaden(material, T; awr=236.0) for T in T_field]
end

# No file I/O in the inner loop
for iteration in 1:max_picard
    T_field = thermal_hydraulics(power_field)
    xs_field = update_neutronics(T_field, material)
    power_field = neutronics(xs_field)
end
```

**Unified data structures:**

```julia
# Same PointwiseMaterial feeds neutronics, heating, and depletion
pendf = reconstruct("u238.endf")
broadened = doppler_broaden(pendf, 600.0; awr=236.0)

# Neutronics: group-averaged cross sections
mg_xs = group_average(broadened.energies, broadened.xs, LANL_30)

# Heating: KERMA coefficients from same data
kerma = compute_kerma(broadened.energies, broadened.xs, material_data)

# Damage: Lindhard partition from same data
damage = elastic_damage(broadened.energies, broadened.xs[:,2], awr)

# All from the same in-memory representation -- no format conversion
```

**Coupled uncertainty propagation:**

```julia
using ForwardDiff

function coupled_response(nuclear_params)
    xs = cross_section_rm(E_grid, params_to_rm(nuclear_params), range)
    broadened = doppler_broaden(E_grid, xs, T; awr=awr)
    power = neutronics(group_average(broadened..., groups))
    T_new = thermal_hydraulics(power)
    fuel_temp = fuel_performance(power, T_new)
    return fuel_temp  # sensitivity of fuel temperature to nuclear data
end

# Gradient through neutronics + T-H + fuel performance
J = ForwardDiff.jacobian(coupled_response, nominal_params)
```

**Estimated impact:** MEDIUM. The multi-physics coupling community is large, but
NJOY.jl's impact depends on the availability of Julia-native neutronics, T-H, and
fuel performance codes. The JENES project at NCSU and broader SciML ecosystem provide
a foundation, but full realization requires a Julia transport code.
